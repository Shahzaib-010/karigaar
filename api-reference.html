API Reference

Everything the frontend needs to talk to the Kaarigar backend — auth flow, every endpoint, request fields, response shapes, and the gotchas that aren't obvious from the response alone.

Base URL & auth

Base URL
http://karrigar-api.test/api (swap host for staging/production)
Auth type
Laravel Sanctum — bearer token
Header
Authorization: Bearer <access_token>
Content type
Accept: application/json on every request (also Content-Type: application/json for JSON bodies)
Token source
returned as access_token from POST /login — store it, attach to every request after

Quick start (register → verify → login → call)

  1. POST /register with name/email/password/phone_number/address → account created as client role, OTP emailed
  2. POST /verify-otp with email + the 6-digit code from the email
  3. POST /login with email/password → get access_token and a redirect_to path telling you which panel to open
  4. Attach Authorization: Bearer <access_token> to every subsequent request
  5. GET /me any time to refresh the current user's profile, roles and permissions (e.g. after a role change)

All four of these are rate-limited per IP: register/login 5 per minute, resend-otp 3 per minute, verify-otp 10 per minute. A 429 means back off and retry after a minute — show a "too many attempts, try again shortly" message rather than treating it as a form error.

Roles & what each can do

Three roles share one users table: client, admin, superadmin. redirect_to from /login tells you which panel to route to — you don't need to inspect roles yourself for routing, but you do need them to conditionally show/hide UI.

Browse categories / sub-categories / pricing✓✓✓

Dev-only seeded logins for testing admin/superadmin panels: admin@kaarigar.com / Admin@12345 and superadmin@kaarigar.com / Super@12345. Registration through /register always creates a client — to make someone an admin, log in as one of these and call PATCH /users/{id}/role.

⚠ Read before building forms

  • status on Category and Sub-Category is boolean (true/false), not a string like "active".
  • Some error messages come back in Urdu (booking slot conflicts, review rules) — treat message as human-readable text to show as-is, don't pattern-match on it in English.
  • Deleting a category, sub-category, price-category, or worker is a soft delete — the record disappears from every list/show endpoint immediately, but existing orders/reviews that reference it keep working and still show its name in their loaded relations.
  • Only a superadmin can grant the superadmin role — an admin calling PATCH /users/{id}/role with role: "superadmin" gets a 403.

One envelope shape for every list

Every list endpoint in this API — categories, sub-categories, price-categories, workers, orders, reviews, notifications, users — returns the same paginator shape, so you can write one generic "unwrap a list response" helper and reuse it everywhere: 

{
  "current_page": 1, "per_page": 20, "total": 46, "last_page": 3,
  "data": [ ... ],
  "first_page_url": "...", "next_page_url": "...", "prev_page_url": "...", "links": {...}
}

Single-item endpoints (show) return {"data": {...}}. Mutations (store/update) return {"message": "...", "data": {...}}. A handful of small, bounded lists that will never need pagination — /roles, /permissions, /workers/available, /booking-slots/available, /booking-slots/calendar — stay as plain {"data": [...]} since wrapping them in pagination metadata would be pure overhead.
